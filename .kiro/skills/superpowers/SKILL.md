# Superpowers

## brainstorm

Use this skill when:

- A task is too large or complex for direct implementation
- Bad architecture is suspected and needs to be addressed before coding
- You need to explore multiple approaches before committing to one
- A conceptually simple task has a surprisingly complex implementation

### Process

1. **Understand the problem** — Restate the goal in your own words. Ask clarifying questions if anything is unclear.
2. **Explore approaches** — Present 2-4 distinct approaches with tradeoffs for each. Include:
   - Pros and cons
   - Complexity estimate
   - Risk factors
   - Your recommendation and why
3. **Wait for user input** — Do not proceed to implementation until the user has chosen an approach.
4. **Break it down** — Once an approach is chosen, decompose it into small, verifiable tasks.
5. **Confirm before coding** — Present the task breakdown and wait for approval before writing any code.

### Rules

- Never start coding during a brainstorm session.
- If you discover a fundamental architecture problem, surface it clearly before proposing solutions.
- Prefer simpler approaches over clever ones unless complexity is justified.
- If the user's preferred approach has significant risks, say so clearly with specific reasons.
